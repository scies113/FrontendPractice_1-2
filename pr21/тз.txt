Действуй как опытный backend-разработчик на Node.js. Твоя задача — доработать существующее серверное приложение на Express (с реализованной ролевой моделью и CRUD-операциями) и внедрить кэширование с использованием Redis[cite: 176, 195].

Шаг 1: Инициализация и настройка
Установи зависимость `redis`[cite: 201]. Настрой подключение к локальному серверу Redis (по умолчанию `redis://127.0.0.1:6379`)[cite: 227].

Шаг 2: Реализация логики кэширования
Создай необходимые функции и middleware для работы с кэшем[cite: 197, 198, 199]:
- `cacheMiddleware(keyBuilder, ttl)`: для проверки наличия нужных данных в Redis перед обращением к БД[cite: 190, 301]. Если данные есть, возвращай ответ клиенту в формате `{ "source": "cache", "data": ... }`[cite: 307, 308, 309].
- `saveToCache(key, data, ttl)`: для сохранения данных в кэш на заданное время, если их там не оказалось[cite: 192, 322]. При отдаче данных напрямую с сервера формат ответа должен быть `{ "source": "server", "data": ... }`[cite: 460, 461].
- Функции очистки кэша (например, `invalidateUsersCache` и `invalidateProductsCache`), которые обязательно должны вызываться при любых изменениях (создании, обновлении, блокировке или удалении) пользователей и товаров, чтобы избежать выдачи устаревших данных клиенту[cite: 547, 548, 549].

Шаг 3: Настройка маршрутов
Примени кэширование к следующим GET-маршрутам со строгим соблюдением времени жизни кэша (TTL)[cite: 588]:
- `GET /api/users` — кэш 1 минута[cite: 588].
- `GET /api/users/:id` — кэш 1 минута[cite: 588].
- `GET /api/products` — кэш 10 минут[cite: 588].
- `GET /api/products/:id` — кэш 10 минут[cite: 588].

Шаг 4: Итоговый результат
1. Предоставь обновленный код приложения. Особое внимание удели интеграции middleware кэширования в роуты чтения и вызовам функций очистки кэша в роутах записи (PUT, POST, DELETE)[cite: 580].
2. Укажи команду Docker для быстрого запуска локального Redis-сервера: `docker run -d --name redis-cache -p 6379:6379 redis`[cite: 584].
3. Напиши краткую инструкцию по тестированию, чтобы убедиться, что при первом запросе `source` равен `server`, а при повторном — `cache`[cite: 555, 569].
